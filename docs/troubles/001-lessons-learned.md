## クライアント主導の冪等性保証による安定性確保

モバイルの特性上、店舗内の通信環境が不安定であるため、APIの通信途切れが頻繁に生じました。  
登録IDをサーバーの自動インクリメントに任せるのではなく、クライアントで生成した一意の `waiting_id` をキーとして送信し、サーバー側でUpsertまたはユニーク制約を適用する「冪等性」を設計したことで、現場での多重クリックによる二重登録バグを完璧に排除できました。

→ [冪等性実装詳細](../implementation/idempotency.md)

## 大容量データ処理時の Isolate 活用による快適なUIの維持

大画面の統計レポートなどで一度に大量の日別データを読み込み、JSONにデコードしてパースする際、パース処理の完了までローディングスピナーのアニメーションが一瞬固まる（Jank）事象がありました.  
Flutter の `compute()` 関数を用いて、重いパース処理をバックグラウンドスレッド(Isolate)で非同期実行させるように改善したところ、60fpsを維持した快適な画面遷移を実現できました。

```dart
// メインUIスレッドでの直実行 (カクつきの原因)
final data = jsonDecode(response.body);

// バックグラウンド Isolate で実行 (解決)
final data = await compute(jsonDecode, response.body);
```

## 各種デバイスのOSネイティブ連携 (カメラおよびプリンター)

1. **QRコードスキャナー**: `mobile_scanner` ライブラリを導入し、 複雑な店舗 ID や ログイン情報を手動で入力することなく、 カメラスキャンひとつで安全に初期設定を完了できるようにし、 店舗の初期設定のハードルを下げました。
2. **整理券プリンター**: 低レベルのBluetooth（ESC/POSコマンド）を叩くのではなく、 `pdf` パッケージで出力紙面を動的にレンダリングし、 `printing` を介して各OS標準の印刷機構（AirPrint、Android Print Service）へ渡す方式を採用。これにより、各メーカー別のプリンターSDKに依存することなく、店舗既存の様々なプリンターをペアリングするだけで容易に出力できる仕組みを構築しました。

## SSEソケット通信と Provider のクリーンな統合

Websocket より軽い HTTP 上でそのままホストされる SSE を Stream API を通じて処理しました。
特に通信途絶等で発生する自動再接続（指数バックオフ）ロジックをサービス層に完全に分離し、UI コンポーネント(ChangeNotifier)はデータの状態のみを受け取るように構造化することで、コードの結合度を下げ、保守性を最大化しました。
