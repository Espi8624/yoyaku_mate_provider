# リアルタイム待機列管理 (Waiting Management)

> 最終更新: 2026-07-10  
> 関連ファイル: [`lib/pages/waiting_page/waiting_screen.dart`](../../lib/pages/waiting_page/waiting_screen.dart), [`lib/pages/waiting_page/waiting_screen_viewmodel.dart`](../../lib/pages/waiting_page/waiting_screen_viewmodel.dart)

## 概要

店舗スタッフが現在待機しているお客様のリストを監視し、呼び出し、入店完了、またはキャンセルといった状態の変化をリアルタイムに指示・操作するコア管理画面です.

---

## 状態フィルタ (タブ構造)

待機リストは、オペレーションの段階に応じて3つのフィルタで切り替えて閲覧できます。
1. **すべて (All)**: 当日のすべての待機履歴。
2. **待機中 (Waiting)**: `waiting` (呼び出し待ち) および `notified` (すでに呼び出したが未入店) のお客様。
3. **完了 (Completed)**: 無事に入店を終えた `completed` 状態のお客様。

---

## 顧客ステータスの操作フロー

スタッフは、リストの各カードに表示されるアクションボタンをワンタップするだけで状況を反映させます。

```
[登録 (waiting)]
       │
       ├─▶ [呼び出し] ボタン ──▶ [呼出済 (notified)] ──▶ お客様のスマホへPush通知/SSEで即時呼び出し
       │
       ├─▶ [完了] ボタン ──▶ [入店完了 (completed)]
       │
       └─▶ [キャンセル] ボタン ──▶ [キャンセル (cancelled)] (モーダルで却下/不在等の理由を選択)
```

### 楽観的UIアップデート (Optimistic Update)

呼び出しや完了などのアクションをタップした際、ネットワーク応答を待ってからUIを切り替えるのではなく、API送信と同時にローカルの状態配列を即時に変更する **楽観的アップデート** を採用。スタッフ側の操作遅延（ラグ）を全く感じさせないサクサクとした応答性を実現しています (`_isPerformingOptimisticUpdate` フラグによる一時的なデータ同期保護)。

---

## スタッフによる手動受付

WebからのQRコードスキャンでの受付以外に, アプリの画面から直接スタッフ自身が口頭で聞いたお客様を代理で登録することも可能です。
- 人数、連絡用電話番号、国籍、個別要望のメモを入力。
- 登録時の通信障害等による重複登録を防ぐため, 事前にクライアント側で生成した一意の冪等性キーを付加して送信します。

→ [クライアント冪等性保証](../implementation/idempotency.md) を参照。

---

## SSE によるリアルタイム自動同期

画面の起動と同時に初期の現在のリストデータをREST APIで取得し、その直後からSSE (Server-Sent Events) のストリーム受信を開始します。これにより、別端末のスタッフが操作したり、お客様が自分のスマホからキャンセルを行った場合でも、手元の画面が一切のリロードなしで自動的にリアルタイムに最新化されます。

→ [SSE購読クライアント実装](../implementation/sse-client.md) を参照。
