# ADR-001: 状態管理ツールとしての Provider パターンの採用

> 作成日: 2026-07-10  
> ステータス: 確定

## コンテキスト

Flutterアプリケーション内において、複数の画面間で共有されるグローバルな状態（認証状態、現在表示中の店舗ID、リアルタイムの待機顧客リストなど）を適切に管理・伝播する手法を検討しました。

---

## 検討対象

| パターン | メリット | デメリット |
|------|------|------|
| BLoC / Cubit | イベント駆動型でデータの状態変化が非常に厳格 | ボイラープレートコードが多く、導入コストが高い |
| GetX | ルーティングと状態管理が一体化し、圧倒的に手軽 | コンテキストを伴わない画面遷移による結合度上昇の危険 |
| Riverpod | コンパイル時のエラーチェック、優れたDI機構 | 記述ルールが独自 (Provider からの移行コスト) |
| **Provider** | Flutter公式推奨歴あり、直感的なMVVMをサポート | 状態肥大化時の再レンダリング最適化に注意が必要 |

---

## 決定: Provider + MVVM パターンの採用

### 理由

1. **シンプルな MVVM 設計の適合**: `ChangeNotifier` を継承した ViewModel (例: `WaitingScreenViewModel`) を作成し、 `ChangeNotifierProvider` を通じて注入する仕組みが最も直感的で冗長な記述を抑えられます。

2. **依存性注入(DI)の容易性**: `MultiProvider` を用いて、 `ProfileService` や `WaitingService` などのAPI通信用サービスレイヤーをアプリケーション最上部 (`main.dart`) から一元的にDIすることが容易です。

3. **低コスト・高効率**: 大規模で超複雑なアプリケーションではなく、単一の店舗に紐づく待機リストを単方向で更新・管理する性質が強いため、標準的な Provider で十分に要件をクリアできます。

### トレードオフ

- 他のProviderの状態に依存して動的に変化させる必要がある場合、 `ChangeNotifierProxyProvider` を経由する必要があり構成がやや複雑化します。
- ウィジェットツリーの階層が深くなった場合に、 `context.read()` や `context.watch()` の無駄な再レンダリングが発生する可能性があるため、描画コストの高い要素（`fl_chart`による統計グラフ等）は `Selector` を活用してリビルドの適用範囲を最適化する必要があります。
